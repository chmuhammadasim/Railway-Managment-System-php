PROJECT CONTEXT: RAILWAY MANAGEMENT SYSTEM (PHP)

1. PROJECT IDENTITY

Project name:
Railway Management System / Pakistan Railways Management System

Project type:
Web-based railway booking and operations management system built as a PHP monolith.

Primary purpose:
This project supports passenger ticket booking, payment, booking updates and cancellation, cargo handling, admin management, and employee railway operations from a single codebase.

Primary user roles:
- Passenger / user
- Employee
- Admin

Main business areas:
- Authentication and profile management
- Public route search
- Seat-based train booking
- Payment handling
- Booking history and ticket details
- Booking cancellation with refunds
- Journey change / booking update
- Cargo shipment handling
- Notifications
- Admin dashboards and CRUD panels
- Employee operations hub
- Waitlist and seat assignment
- Audit logging
- Password reset by email
- E-ticket email sending


2. HIGH-LEVEL TECH STACK

Backend:
- PHP 8 style code
- No full framework such as Laravel, Symfony, or CodeIgniter
- Mostly procedural page controllers plus a few reusable classes
- MySQLi for database access

Database:
- MySQL / MariaDB database named `railway_system`
- SQL schema lives in `database/railway_schema.sql`
- Demo/seed data is included in the schema file

Frontend:
- Server-rendered PHP + HTML
- Bootstrap 5 from CDN
- Bootstrap Icons from CDN
- Google Fonts from CDN
- Custom CSS in `public/css/style.css`
- Several pages also contain large inline CSS blocks
- Small vanilla JavaScript helpers in `public/js`

Mail:
- PHPMailer is vendored directly in `src/PHPMailer`
- SMTP settings are loaded from `config/mail.php`

Runtime environment:
- Intended for XAMPP on Windows
- Apache + MySQL
- Repository located under `C:\xampp\htdocs\Railway-Managment-System-php`


3. ARCHITECTURAL STYLE

This project is not API-first and not component-driven. It is a classic PHP application where:
- Root-level `.php` files act as page controllers and views
- Each page manually includes config and required classes
- Business logic is partly inside pages and partly inside classes under `src/classes`
- Shared layout is included through `inc/header.php` and `inc/footer.php`
- Pages perform access control checks directly
- Pages often query the database directly for dashboards and listings
- Reusable workflows such as booking, payments, operations, logging, and password reset are wrapped in classes

Important architectural characteristics:
- No Composer setup was found
- No routing layer was found; URLs map directly to physical PHP files
- No migrations framework was found; schema is mainly SQL-file based, with a few runtime safety migrations in PHP
- No template engine was found; views are raw PHP templates
- No asset build step was found; CSS and JS are served directly

5. CORE BOOTSTRAP PATTERN USED BY MOST PAGES

Most pages follow this pattern:

1. Include config and required classes:
- `require_once 'config/database.php';`
- `require_once 'src/classes/Database.php';`
- `require_once 'src/classes/User.php';`
- then any page-specific class files

2. Enforce authentication or role access:
- `User::isLoggedIn()`
- check `$_SESSION['role']`

3. Create and connect the database wrapper:
- `$db = new Database();`
- `$db->connect();`

4. Handle POST actions or GET filters

5. Query data and/or call business classes

6. Set page variables:
- `$pageTitle`
- sometimes `$hideMainNavbar`
- sometimes `$extraScripts`

7. Render:
- `require_once 'inc/header.php';`
- page HTML/CSS/JS
- `require_once 'inc/footer.php';`

7. DATABASE WRAPPER CONTEXT

`src/classes/Database.php` is a thin MySQLi wrapper.

Responsibilities:
- connect using constants from `config/database.php`
- set charset to `utf8`
- expose helper methods:
  - `select()`
  - `selectRow()`
  - `insert()`
  - `update()`
  - `delete()`
  - `query()`
  - `countRows()`
  - `getConnection()`
  - `getLastId()`
  - `getError()`

Important runtime behavior:
- `connect()` runs `ensureCancellationColumns()`
- that method checks `information_schema` and adds missing booking cancellation columns if needed
- this was added because some SQL syntax in the schema is MariaDB-friendly and not always safe on MySQL 8

9. MAJOR USER FLOWS

Public browsing flow:
- `index.php` loads upcoming/scheduled routes
- filters by departure city, arrival city, and journey date
- shows counts such as active trains and confirmed bookings

Authentication flow:
- `signup.php` creates passenger accounts
- `login.php` accepts username or email + password
- successful login redirects to:
  - `dashboard.php`, then
  - `admin-dashboard.php` for admins
  - `employee-dashboard.php` for employees
  - passenger dashboard for users

Booking flow:
- user opens `book.php?route_id=...`
- if route has no seats yet, seats are auto-generated
- page renders seat map grouped by class
- user selects up to 6 seats
- passenger details are submitted
- `Booking::createBooking()` runs in a transaction
- booking is created as pending until payment completes
- user is redirected to payment

Payment flow:
- `payment.php` simulates multiple payment methods
- `Payment::processPayment()` inserts a payment row
- booking becomes confirmed and payment status becomes completed

Booking details flow:
- `booking_details.php` shows train, route, seats, passenger, and payment data
- can expose actions such as update, cancel, or pay

Cancellation flow:
- refund preview is based on hours before departure
- unpaid reservation can be canceled without fee
- paid booking may be refunded depending on timing
- seats are released
- route available seat count is synced
- waitlist processing can be triggered

Journey change flow:
- booking can move to another route with same origin/destination
- system tries to preserve seat-class mix
- if new route costs more, additional payment is required
- if new route costs less or equal, booking may confirm without extra charge

Cargo flow:
- passenger can create cargo shipment
- shipment types include delivery and traveling-with-cargo
- tracking number is generated
- admin/staff can manage shipment lifecycle

Operations flow:
- `operations-hub.php` is the central operational page
- admins and employees can work with operational modules
- this page explicitly creates and verifies a CSRF token

Email flows:
- password reset email via `PasswordReset.php`
- e-ticket email via `send_ticket_email.php`

Real-time / AJAX flow:
- `api/seat-availability.php` returns seat data as JSON for a route or a single seat
- `update_train_status.php` updates route status and corresponding live train status

14. KNOWN IMPLEMENTATION NOTES AND QUIRKS

Important quirks observed from the codebase:

- The app is meant to run from `http://localhost/Railway-Managment-System-php/`, but `SITE_URL` in `config/database.php` points to `http://localhost/railway/`.

- `Database::connect()` only auto-fixes cancellation columns on `bookings`; operational tables are created by `Operations::ensureSchema()` and password reset table is created by `PasswordReset::ensureTable()`.

- `book.php` computes class-based fare multipliers in the page, but `Booking::createBooking()` itself starts with `base_fare * seat_count`. If you extend fare logic, review both places together.

- The project mixes direct page SQL and class methods. This means changes to a business rule may need edits in multiple files, not only in one class.

- PHPMailer is vendored locally; there is no Composer autoload flow.

- Pages rely on physical file routing, so file names are part of the app's public contract.
